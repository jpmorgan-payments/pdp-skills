Demos

Watch the skills build and migrate working ecommerce integrations, end to end, in minutes.

The demos below are grouped into two tracks. Integrations covers merchants coming to J.P. Morgan from the outside — a greenfield build from an empty project, or a migration off a third-party gateway like Stripe. Orbital Migrations covers merchants already running on a legacy JPM product (Orbital or HPS) who are moving to a modern JPM Payments API. In every case the skills drive the conversation, generate the OAuth module, and wire up the target JPM Payments API.

What you're seeing: a project opened in an Agent harness, driven entirely by the skills' conversational flow. The merchant types one short prompt to start; everything after is multiple-choice answers and the skill generating code in the merchant's project. The generated code is a starting point — every file should be reviewed by a qualified engineer before being deployed (see limitations & disclaimers).

The Integrations track is for merchants coming to J.P. Morgan from outside — whether they're starting from an empty codebase or migrating off a third-party gateway. The skills confirm credentials, generate the OAuth module, and wire up the target JPM Payments API, so the merchant can stand up a working integration and validate in CAT in minutes.

Greenfield Merchandise Store

Greenfield Merchandise / D2C apparel Checkout Google Pay

A direct-to-consumer brand launching a branded merch store wants to start selling online fast. They've finished the paperwork side of JPM and have credentials, but their codebase is empty. They open their Agent harness, invoke /jpm-integrations-get-started, and the skills do the rest — confirming credentials, generating an OAuth module with token caching, scaffolding the storefront, and integrating JPM Checkout with Google Pay.

Why it matters: what would normally take a small engineering team several days — reading API docs, picking libraries, getting auth right, wiring Checkout, smoke-testing in CAT — happens in a few minutes of guided conversation. The merchant focuses on their brand and product; the skills handle the JPM API surface.

Brownfield Stripe Migration

Migration Stripe → JPM Merchandise / D2C apparel Checkout

A merchant already selling online through Stripe wants to move their payments to J.P. Morgan without rebuilding their store. They have an existing codebase and a working checkout page — the skills guide them through generating OAuth and swapping the Stripe payment flow for JPM Checkout, mapping the concepts they already know from Stripe Elements onto the JPM API surface. The rest of the storefront stays exactly as it is.

Why it matters: moving off an incumbent gateway is usually a manual, risky rewrite. The skills encode JPM's documented patterns and generate the OAuth and Checkout wiring in the merchant's existing project, so switching processors becomes a guided change instead of a multi-week project. The output is a starting point, not production code — every generated file should be reviewed by a qualified engineer before deployment (see limitations & disclaimers).

The Orbital Migrations track is for merchants already running on a legacy J.P. Morgan product — the Orbital gateway or the Hosted Payments Solution (HPS) — who want to move to a modern JPM Payments API. Rather than a rip-and-replace rewrite, the skills map the concepts the merchant already knows from the legacy product onto the new API surface and generate the OAuth and integration wiring.

Orbital to Online Payments Migration

Migration Orbital → Online Payments Online Payments

A merchant running on the legacy Orbital gateway wants to move to JPM's modern Online Payments API. Rather than a rip-and-replace rewrite, the skills guide them through generating OAuth and re-wiring their payment flow to the Online Payments endpoints — mapping the concepts they already know from Orbital onto the new API surface. Same conversational experience, focused on migration instead of greenfield.

Why it matters: migrations off legacy gateways are usually slow, risky, and manual. The skills encode the target API's documented patterns and generate the OAuth and Online Payments wiring, so the merchant can move off Orbital and validate in CAT in minutes. The output is a starting point, not production code — every generated file should be reviewed by a qualified engineer before deployment (see limitations & disclaimers).

HPS to Checkout Migration

Migration Hosted Payments → Checkout Checkout

A merchant on the legacy Hosted Payments Solution (HPS) moving to JPM Checkout. The skills walk them through generating OAuth and integrating Checkout into their existing storefront, replacing the older hosted-payment flow with the modern Checkout experience — including wallet options like Google Pay. Different legacy product, same guided path to a modern integration.

Why it matters: the skills aren't tied to a single starting point or product. Greenfield builds and legacy migrations (Orbital, HPS, and others) all run through the same conversational flow. The skills' scope is limited to integrating JPM Payments APIs into the merchant's project; they don't build the rest of the business logic.

What These Demos Show

  • Time-to-first-CAT-transaction collapses from days to minutes. No reading the JPM API reference cover-to-cover, no copy-pasting OAuth boilerplate, no field-constraint trial and error.
  • Minimal prompting. The merchant types one short prompt to kick off the conversation; the skill drives the rest with multiple-choice questions.
  • Best-practice patterns by default. The generated OAuth module caches access tokens with an in-flight mutex (avoiding the per-request token regen anti-pattern), and the Checkout integration imports the auth module rather than re-implementing it. The output is a starting point, not production code — every generated file must be reviewed by a qualified engineer before deployment. See the limitations & disclaimers.
  • Works across more than the demos shown. D2C merchandise and retail are two examples — the skills don't hard-code a vertical and adapt to what the merchant describes (subscription commerce, marketplaces, B2B, ticketing, donations, services, and others). Scope: integrating JPMorgan Payments APIs into the merchant's project. The skills don't build the rest of the business logic — that's still the merchant's work.
  • The merchant stays in their own codebase. Nothing leaves their machine. Credentials stay in their .env.
